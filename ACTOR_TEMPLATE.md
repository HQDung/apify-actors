# Actor Template Operating Guide

## Niche JSON requirements

Each `niches/*.json` file is the source of truth for generated Actor metadata and inputs. It must include:

- `actorName`, `actorTitle`, `niche`, `nichePlural`, `defaultKeyword`, `defaultLocation`
- `shortDescription`, `longDescription`, `targetUser`, `buyerPainPoint`, and `differentiation`
- an object-valued `sampleInput`
- `benchmarkInputs` entries labelled `smoke`, `email-baseline`, and `broader-search`, each with an object-valued `input`

Optional `multiSearch` metadata describes niche-specific location, type, and result-limit fields. Actor names must be lowercase hyphenated names suitable for a directory and Apify Actor name.

## Generated files

The factory copies the template into a standalone directory and renders the niche metadata into:

- `.actor/` configuration and schemas
- `README.md`, including the rendered sample and benchmark inputs
- `storage/key_value_stores/default/INPUT.json`
- `src/niche-config.js` and the template runtime modules
- the template tests and project configuration

The existing `actors/` directories are not regenerated by this workflow.

## Standalone deploy rule

Generated Actors must run and deploy from their own directory. Validate first, then use `apify run` or `apify push` from `actors/<actor-name>`; the Actor must not import runtime files from the factory root. Publishing is always an explicit, selective action and is never automatic.

## Validation

Run the static contract validator before any benchmark or publish decision:

```bash
npm run validate:actor -- actors/vietnam-spa-lead-scraper
```

It checks required generated files, JSON, unresolved factory markers, and README dataset-field coverage.

## Benchmark workflow

1. Select a niche and one labelled input from its `benchmarkInputs`.
2. Validate the generated Actor.
3. Run a local benchmark with `npm run benchmark -- --actor-dir actors/<actor-name> --niche-file niches/<niche>.json --label smoke`.
4. Review the local dataset counts and the appended row in `BENCHMARKS.md`.
5. Treat cloud testing as a separate step; local cost is recorded as `local-only / not available`.

Use `--dry-run` to resolve and inspect an input without running the Actor or writing benchmark output.

## Publishing policy

Do not auto-publish and do not auto-change pricing. Cloud tests and selective publishing require a separate explicit decision after validation and review.
